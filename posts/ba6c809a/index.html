产品软件版本管理

Overview

基于开源工程开发产品，应该以自身产品为主导。产品有自己的软件库，自己的固件版本，企业的产品应该有良好的版本控制，对于用户或者企业内部开发人员，都能很好地跟踪不同版本 changes 或者 bug。

下面以基于 LEDE 开源工程开发路由器固件为切入点，记录我在工作中基于开源工程开发产品的版本管理。

release vs. trunk

LEDE Source

LEDE git repo: 

https://git.lede-project.org/source.git

LEDE Branch

LEDE git repo 有两个分支：lede-17.01 和 master。其中 master 分支为开发分支，每时每刻都在更新，俗称 “trunk” 分支；lede17.01 为发布分支，一般只有重大的安全问题才会更新，俗称 “release”。

Snapshots vs. Releases

LEDE Firmware Repository 有两个不同的分支：Stable Release builds 和 Development Snapshot builds。Downloads： 

https://downloads.lede-project.org/releases/
https://downloads.lede-project.org/snapshots/

Initiate LEDE Release Repo

Download Source

$ git clone https://git.lede-project.org/source.git

View Release Info

$ git tag -l
reboot
v17.01.0
v17.01.0-rc1
v17.01.0-rc2
v17.01.1
v17.01.2
v17.01.3
v17.01.4

Checkout to Release

$ git branch -D lede-17.01
$ git checkout -b lede-17.01 v17.01.4

Replace with Owner Repo URL

$ git rm .git/config
$ git remote add origin https://github.com/gl-inet/lede-17.01

Push Release

$ git push origin lede-17.01

Commit Changes

建议创建另一个开发分支进行开发，然后 merge 到 lede17.01.4 分支上。开发工作完成后，我们需要发布一个固件，版本为 v1.0。由于 LEDE 的 feeds 是不断更新，或者私有软件包也不断的更新，我们需要记录 v1.0 发布时所用的 feeds 或者 私有软件包的对应版本，一般都是用 git 管理，所以我们需要记录 commit 值。先提交，然后 release，再回退。 

$ git commit -m "adjust config defaults"
$ git push origin lede17.01

$ git tag v1.0 -m "GL-iNet v1.0 Release"
$ git push origin v1.0  

$ git commit -m "revert to branch defaults" 
$ git push origin lede17.01

Sync Original Source

LEDE 不断地更新，稳定版本升级到了 lede17.01.8，更新了重要的安全补丁，我们需要把它同步到我们的工程。 

$ git checkout lede-17.01
$ git remote add upstream https://github.com/lede-project/source.git
$ git fetch --prune --tags  upstream
$ git merge v17.01.8
$ git push origin lede17.01
$ git remote remove upstream

这个过程很有可能有代码冲突，需要手动解决冲突。

Note

个人觉得软件版本管理对于企业来说，是很重要的。如果每个版本都有记录，我们可以看到每个版本都修改了些什么内容，以便 bug 追踪，也方便版本回退。
